## 核心主題

本 Lab 從「規則記憶體存取」跳轉到 **「不規則記憶體存取」**（Irregular Memory Access Patterns）——這是許多真實應用的特性。數據依賴無法預測，記憶體存取不連貫，這正是並行計算的難點。

## 性能目標

| Part | 任務 | 目標 | 測試規模 |
|------|------|------|---------|
| **0** | Warp Shuffle vs Shared Memory | 性能對比分析 | 32 元素 |
| **1** | Parallel Scan | ≥144 GB/s（40% 峰值） | 2²⁶ (~67M) 元素 |
| **2** | RLE 壓縮 | ≤1.6 ms | ~16MB 檔案 |
| **3** | RLE 解壓縮 | ≤0.8 ms | 動態分配 |

## 這個 Lab 在做什麼

做了一個 GPU 上的 Run-Length Encoding 壓縮系統。簡單說就是把連續重複的值壓縮成「值 + 重複次數」的格式，然後用 warp shuffle 和並行掃描來加速。

核心技術：Warp Shuffle（warp 內零延遲通信）+ Parallel Scan（分層掃描）

## Part 0: Warp Shuffle 實驗

### 問題

實現一個 32 元素的累積和，用兩種方法比較：
1. 用共享記憶體
2. 用 warp shuffle

### 發現有多快

| 方法 | 時鐘週期 | 快多少 |
|------|---------|--------|
| 共享記憶體 | 1659 | 基準 |
| **Warp Shuffle** | **264** | **6.28× 快** |

這結果有點驚人。預期是 3-5 倍，結果直接快了 6 倍多。

### 為什麼差這麼多？

**共享記憶體版的問題**：
- 每次迭代都要讀記憶體、同步、寫記憶體、再同步
- 迴圈 5 次（log₂32），每次都是雙重同步開銷
- L1/L2 cache 競爭，有 bank conflict

**Warp Shuffle 的妙處**：
- Warp 內的 32 個 thread 天生同步（lockstep 執行）
- 用 `__shfl_up_sync()` 直接在寄存器間傳值
- 沒有共享記憶體讀寫，沒有同步等待
- 硬體級原生支持，延遲超低

### 實際感受

用 shuffle 時才意識到 GPU 的設計有多精妙。warp 是 GPU 的基本執行單位，天生就是「全部同時做同一件事」。既然 32 個 thread 已經鎖步了，何必還要用共享記憶體和顯式同步來模擬？直接讓硬體幫你傳值就完事。

## Part 1: Parallel Scan（並行掃描）

### 核心想法

要對 N 個元素做前綴和，不能 block 級別搞一搞就結束，因為 N 可能很大（幾百萬）。所以分三層：

```
層級 1：Block 內掃描（256 個 thread，共享記憶體 + warp shuffle）
  └─ 每個 block 輸出一個聚合值（最後元素的累積和）

層級 2：掃描所有 block 的聚合值
  └─ 得到每個 block 應該加上的「起始偏移」

層級 3：Fix-up（加回偏移）
  └─ 每個 block 的所有元素加上它的偏移，完成全域掃描
```

### Block 內掃描怎麼做

每個 block 有 8 個 warp（256 thread）。策略是：

1. **Warp 內掃描**（用 warp shuffle）
   - 每個 warp 內部的 32 個 element 用 shuffle 掃一次
   - 最快，無同步開銷

2. **聚合 8 個 warp 結果**
   - 提取每個 warp 最後的累積值
   - 對這 8 個數字做掃描
   - 得到每個 warp 應該加的偏移

3. **分發偏移**
   - 把每個 warp 的偏移廣播給它的 32 個 thread
   - 每個 thread 把自己的值加上偏移

這樣做的好處是充分利用了 warp shuffle 的快速，又用共享記憶體處理 block 級同步。

### 目標效能

要達到 ≥144 GB/s（RTX 4000 Ada 峰值 360 GB/s 的 40%）

我覺得 40% 的目標很實際。記憶體 bound 的算法很難超過 60-70%，而掃描涉及很多全域記憶體訪問，有同步開銷。能達到 40% 就是相當不錯的實現了。

### 最容易踩的坑

**Warp-level vs Block-level**：一開始容易搞混 warp 內和 block 內的同步。warp 內無需同步（lockstep），block 內需要 `__syncthreads()`。

**跨 block 通信**：不同 block 的 thread 無法直接同步，所以必須分成多個 kernel 或用 global barrier。這是設計上的限制。

**邊界情況**：當數據大小不是 256 的倍數時，最後一個 block 要特殊處理。

### 實現細節

做掃描時發現幾個小陷阱：

**標記邊界**：用輔助陣列記錄 run 的起始位置。每個 thread 比較自己和前一個元素，產生一個布林值。這個布林陣列隱含的結構很重要——它決定了後續掃描的步驟。

**動態共享記憶體**：因為題目限制不能用 CUB 庫，所以手寫共享記憶體的分配。要小心的是，C++ 中動態共享記憶體得靠 char* 指針來 cast。`__shared__ char sm[];` 然後手動計算偏移。一開始踩過坑，不小心 offset 算錯導致記憶體亂寫。

**測試規模**：實驗用 2²⁶（約 67 百萬）元素。在這個規模下，掃描的開銷才顯著。小規模下看不出效果。

## Part 2: RLE 壓縮

### 實際問題

有個像素陣列，很多連續重複的值。要把它們壓縮成「值 + 重複次數」的格式。

例如：
```
輸入：[255, 255, 255, 0, 0, 128, 128, 128, 128]
輸出：[255, 3, 0, 2, 128, 4]
```

### 怎麼並行化

這個問題的妙處在於單靠 thread 本身解決不了，因為你不知道「這個值要重複幾次」，必須看後面的元素。所以需要掃描。

**步驟 1：找邊界**
```
比較相鄰元素。如果 A[i] ≠ A[i-1]，就是新 run 的開始
例：[255, 255, 255, 0, 0, 128, ...]
邊界：[1, 0, 0, 1, 0, 1, ...]  ← 標記 run 的起始位置
```

**步驟 2：掃描邊界**
```
對邊界標記做前綴和，得到「run ID」
邊界：    [1, 0, 0, 1, 0, 1, ...]
Run ID：  [1, 1, 1, 2, 2, 3, ...]
```

**步驟 3：Compaction**
```
只有邊界位置的 thread 才執行輸出：
- 輸出當前的像素值
- 計算 run 長度（下一個 run 開始的位置 - 當前位置）
- 寫入輸出緩衝區
```

### 為什麼要這樣設計

直覺上似乎可以「每個 thread 自己算自己的」，但實際上做不到。因為：
- 你不知道你的值重複多少次（取決於後面的數據）
- 輸出位置也不固定（取決於前面有多少個 run）

所以必須用掃描這種「全局視角」的操作。這是並行算法的典型：局部操作解決不了的，用全局掃描轉化成可並行的形式。

### 效能考量

**讀取**：連續讀輸入，很快
**寫入**：稀疏寫輸出，取決於 run 密度。若大部分像素都不同（高熵），輸出量接近輸入，壓縮率差。若很多重複，輸出量小。

記憶體頻寬利用率完全靠數據的 run 長度分佈。

### 實現細節

這部分的優化空間在「邊界標記」和「輸出策略」：

**邊界標記的並行化**：每個 thread 獨立比較自己和前一個元素。第一個 thread 總是標記為 1（run 開始）。要小心邊界：最後一個元素的下一個不存在，不能超出邊界讀。

**稀疏輸出的位置計算**：用掃描得到 run ID 後，只有 run 邊界的 thread 才寫輸出。輸出位置可以透過掃描結果直接計算。這裡最容易錯的是：計算 run 長度時，不能假設下一個 run 的位置已知，因為有些 thread 根本不執行輸出。

**測試規模**：~16MB 的檔案。在這個大小下，I/O 成本不是主要瓶頸，重點是算法正確性和記憶體存取效率。

## Part 3: RLE 解壓（選作）

### 反向操作

有一個壓縮的陣列，要展開成原始像素。

```
輸入：[255, 3, 0, 2, 128, 4]
輸出：[255, 255, 255, 0, 0, 128, 128, 128, 128]
```

### 思路

1. 每對 (value, count) 計算在輸出中的起始位置
2. 多個 thread 並行寫這個 run 的每個像素

例如 (255, 3) 對應輸出位置 [0, 1, 2]，可以用 3 個 thread 並行寫。

長 run 可以分配給多個 thread，充分利用並行性。

### 實現細節

解壓比壓縮更有意思，因為輸出大小預先不知道：

**動態記憶體分配**：輸出大小取決於所有 run 的 count 總和。題目要求用 `GpuAllocCache` 來管理臨時記憶體。第一次掃描輸入得到總大小，再動態分配輸出陣列。

**並行 scatter 寫**：每對 (value, count) 可以由多個 thread 並行處理。如果 count=1000，可以用 1000 個 thread 並行寫 1000 個像素。這要靠前綴掃描得到每對的起始位置。

**負載平衡**：如果有些 run 很長（比如 count=10000）而有些很短（count=1），會造成執行時間不均。一個簡單的策略是：每個 thread 寫自己負責的位置範圍。複雜一點的方案能用 warp-level 同步來進一步優化。

**目標**：≤0.8 ms 完成解壓。這對百萬級規模的輸出很有挑戰。

## 實際做下來的體會

### Warp Shuffle 改變了什麼

之前都是「共享記憶體 + 同步」的思維，才意識到 warp 級別還有另一套玩法。一旦理解了 lockstep 執行，就能充分利用硬體特性。6 倍的性能差異說明了什麼是「跟硬體的設計思想一致」。

### Parallel Scan 是個好例子

掃描看起來簡單，但實現時牽涉很多層次的考慮：
- Warp 級（無同步）
- Block 級（`__syncthreads()`）
- 全域（多 kernel 或 global barrier）

每層各有各的同步機制和性能特性。這教會了我並行算法不是「寫成循環就行」，細節差很大。

### RLE 這個應用挺有意思

壓縮不像矩陣乘法那樣「計算密集」，反而是「數據依賴密集」。必須用掃描這類技巧把依賴關係打散，才能並行。這讓我理解了為什麼掃描是並行計算的基本操作——它解決的是「怎樣從局部依賴推導出全局狀態」這個問題。

## 做這個 Lab 學到的

 **Warp 是執行的最小單位**
- 32 個 thread 天生鎖步
- 可以用 shuffle 零成本地傳值
- 比共享記憶體快得多

 **並行掃描不只是「一個算法」**
- 它是分層的
- 每層同步機制不同
- 用得好能達到 40% 峰值效率

 **算法和硬體設計要相符**
- 如果你逆著硬體設計走（比如不必要的同步），性能會慘
- 如果你跟著硬體走（比如 warp shuffle），能快好幾倍

 **並行化不是加個並行符號就行**
- 有些問題需要掃描、聚集這類全局操作才能破
- 需要理解數據依賴，選對算法
